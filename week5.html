What is the Raspberry Pi Pico W?

The Raspberry Pi Pico W is the wireless-enabled variant of the original Pico, incorporating an Infineon CYW43439 module that provides 802.11n Wi-Fi (2.4GHz) and Bluetooth 5.0 (Bluetooth Classic and LE). This addition transforms the Pico from a standalone microcontroller into a network-connected device capable of participating in IoT ecosystems.

Key Differences from Pico (Non-W)

  • Wireless Module: CYW43439 provides Wi-Fi and Bluetooth connectivity
  • LED Behavior: Onboard LED (GPIO 29) is now connected to the Wi-Fi module
  • Power Consumption: Higher current draw when Wi-Fi is active
  • Form Factor: Identical pinout — Pico W is a drop-in replacement

Wi-Fi Configuration and Initialization

Connecting to wireless networks requires specific MicroPython modules and proper configuration.

Network Module

The network module provides the interface for Wi-Fi operations:

import network
import time

# Create Wi-Fi interface
wlan = network.WLAN(network.STA_IF)

# Activate the interface
wlan.active(True)

# Scan for available networks
networks = wlan.scan()

# Connect to a network
wlan.connect('SSID', 'password')

# Wait for connection
while not wlan.isconnected():
    time.sleep(1)
    
print('Connected:', wlan.ifconfig())

Wi-Fi Modes

  • Station (STA): Connect to an existing Wi-Fi network (most common for IoT)
  • Access Point (AP): Create a Wi-Fi network for other devices to connect to

Connection Parameters

  • SSID: The network name (Service Set Identifier)
  • Password: WPA2/WPA3 security credential
  • Channel: Wi-Fi channel (1-11 for 2.4GHz)
  • Security: WPA2-PSK or WPA3-PSK

Building a Web Server on Pico W

A web server enables remote interaction with the Pico W through any web browser — providing a user interface without dedicated applications.

What is a Web Server?

A web server is software that listens for HTTP (Hypertext Transfer Protocol) requests from clients (web browsers) and responds with appropriate data (HTML pages, JSON, status codes). On the Pico W, we implement a lightweight HTTP server using MicroPython's socket module.

How Web Servers Work

  1. Client Request: A browser sends an HTTP request to the server's IP address
  2. Request Parsing: The server parses the request method (GET, POST) and path
  3. Processing: The server executes appropriate logic based on the request
  4. Response Generation: The server constructs an HTTP response with status code and content
  5. Transmission: The response is sent back to the client
  6. Rendering: The browser displays the received content

Implementation Steps

1. Create a Socket

import socket

# Create TCP socket
addr = socket.getaddrinfo('0.0.0.0', 80)[0][-1]
s = socket.socket()
s.bind(addr)
s.listen(1)

2. Handle Requests

while True:
    conn, addr = s.accept()
    request = conn.recv(1024)
    request = str(request)
    
    # Parse the request path
    if '/led/on' in request:

Debugging Techniques

Effective debugging is essential for developing reliable embedded systems. We employed multiple strategies:

Serial Output Debugging

  • Print Statements: Output variable values and program state via UART
  • REPL Access: Use the MicroPython REPL for interactive debugging
  • Exception Handling: Catch and display errors to identify issues

LED Indicator Debugging

  • Status Indication: Blink patterns to indicate program state
  • Error Codes: Different blink sequences for different error conditions
  • Progress Indication: Blink during initialization phases

Network Debugging

  • Connection Status: Print Wi-Fi connection state during startup

Hardware Debugging Tools

Advanced Wi-Fi Features

Beyond basic connectivity, we explored advanced networking capabilities:

Access Point Mode

The Pico W can function as a Wi-Fi access point, creating its own network:

ap.active(True)
ap.config(essid='PicoW-Setup', password='password')

The Pico W can act as an HTTP client to send data to cloud services:

MQTT Protocol

For IoT applications, MQTT provides lightweight messaging:

  • Publish: Send sensor data to a broker
  • Subscribe: Receive commands from a broker
  • Broker: Central server that routes messages (e.g., Mosquitto, HiveMQ)

Security Considerations

Network-connected devices require careful security attention:

  • Wi-Fi Security: Use WPA2/WPA3 — never open networks
  • Input Validation: Sanitize all HTTP request parameters
  • Credential Storage: Never hardcode passwords in source code
  • Updates: Keep firmware updated for security patches
  • Firewalls: Restrict network access to necessary services only

Push Button LED Control

Beyond network-based control, we implemented physical input handling through push buttons — a fundamental interaction paradigm in embedded systems.

Debouncing Algorithm

Mechanical push buttons exhibit a phenomenon known as "contact bounce" — when the button is pressed or released, the contacts physically bounce, causing multiple rapid transitions that can be misinterpreted as multiple button presses. To mitigate this, we implemented a debouncing algorithm:

  • State Monitoring: Continuously read the button's GPIO state
  • Time-Based Filtering: Require the button state to remain stable for a defined period before accepting the input
  • Edge Detection: Detect only transitions (pressed or released) rather than continuous states

Toggle Functionality

We implemented a toggle mechanism where each button press alternates the LED state:

  • First Press: The onboard LED illuminates — indicating an "ON" state
  • Second Press: The onboard LED extinguishes — indicating an "OFF" state
  • Subsequent Presses: The cycle repeats, creating a reliable binary state control

This exercise demonstrated the intersection of input handling, state management, and output control — the foundational pattern upon which all embedded user interfaces are built.

HC-SR04 Ultrasonic Sensor

We interfaced the HC-SR04 ultrasonic ranging sensor with the Pico W to measure distance to objects — a sensor commonly employed in obstacle detection, robotics navigation, and proximity sensing applications.

How Ultrasonic Ranging Works

The HC-SR04 utilizes the time-of-flight principle to determine distance:

  • Trigger Pulse: The microcontroller sends a 10-microsecond trigger pulse to the sensor
  • Ultrasonic Emission: The sensor emits an 8-cycle ultrasonic burst at 40kHz
  • Echo Detection: The sensor listens for the reflected sound wave
  • Time Measurement: The duration between emission and echo reception is measured
  • Distance Calculation: Using the speed of sound (approximately 343 m/s), distance is computed as: Distance = (Time × Speed of Sound) / 2

Distance Measurement Implementation

We configured the Pico W to:

  • Send trigger pulses at regular intervals
  • Measure the echo pulse width using timer inputs
  • Calculate distance in centimeters based on the time measurement
  • Print the measured distance to the serial console for real-time monitoring

Personal Project: Proximity Alert System

Extending the ultrasonic measurement capability, I developed a comprehensive proximity alert system that integrates multiple output modalities to provide intuitive distance feedback.

System Components

  • HC-SR04 Ultrasonic Sensor: Measures distance to nearby objects
  • I2C LCD Display: 16×2 character display shows distance readings and status messages
  • Piezoelectric Buzzer: Emits audible warnings at configurable intervals
  • Onboard LED: Provides visual indication of danger states

Operational Logic

The system implements a tiered alert mechanism based on measured distance:

Safe Zone (Distance > 20 cm)

  • The LCD displays "SAFE" along with the measured distance
  • The buzzer remains silent
  • The onboard LED is extinguished

Warning Zone (Distance ≤ 20 cm)

  • The LCD displays "WARNING" along with the measured distance
  • The buzzer emits periodic beeps at a moderate rate
  • The onboard LED remains off

Danger Zone (Distance < 15 cm)

  • The LCD displays "DANGER" along with the measured distance
  • The buzzer increases its beep rate significantly — indicating imminent collision
  • The onboard LED illuminates continuously — providing visual warning

Project Demonstration

This project exemplifies the integration of multiple subsystems — input sensing, conditional logic, and multi-modal output — into a cohesive embedded application. The combination of visual, auditory, and textual feedback ensures that users receive clear, unambiguous information regardless of their sensory capabilities or environmental conditions.
