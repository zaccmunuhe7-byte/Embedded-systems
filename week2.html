Week 2 - C Programming & Assembly Language

Overview

Week 2 marked a pivotal transition from high-level web technologies to the foundational programming paradigms that underpin all embedded systems development. While HTML and CSS serve as the visible interface through which we document our journey, C and Assembly Language constitute the invisible machinery that animates microcontrollers, orchestrates sensor data, and executes real-time control algorithms. This week endowed us with the intellectual tools necessary to communicate directly with silicon — translating algorithmic intent into electrical signals that govern physical systems.

C Programming Language

C has reigned supreme as the lingua franca of embedded systems development for over five decades. Its enduring prominence stems from an unparalleled combination of low-level hardware access and cross-platform portability. Unlike high-level languages that abstract away hardware details, C permits direct manipulation of memory addresses, register values, and hardware peripherals — capabilities essential when programming resource-constrained devices such as the Raspberry Pi Pico, Arduino, or STM32 microcontrollers.

Variables and Data Types

Variables serve as named containers for storing data within program memory. In embedded C, selecting the appropriate data type is paramount — each type consumes a specific amount of memory (measured in bytes), and on devices with limited RAM (often 64KB to 256KB), every byte matters.

  • char: A single byte (8 bits) capable of storing ASCII characters or integers from -128 to 127. Commonly used for storing character data and small numerical values.
  • int: Typically 2 or 4 bytes, depending on architecture, storing integers ranging from -32,768 to 32,767 (16-bit) or -2,147,483,648 to 2,147,483,647 (32-bit). The workhorse of embedded calculations.
  • float: 4 bytes representing floating-point numbers with approximately 6-7 significant digits. Essential for sensor readings and mathematical computations requiring decimal precision.
  • double: 8 bytes providing greater precision (15-16 significant digits) for complex scientific calculations, though rarely used in memory-constrained embedded applications.
  • unsigned variants: Versions of the above (e.g., unsigned int) that exclude negative values, effectively doubling the maximum positive range.

Control Structures

Control structures dictate the flow of program execution, enabling decision-making and iteration — capabilities fundamental to embedded applications such as monitoring sensor thresholds or polling hardware states.

  • if/else statements: Conditional execution blocks that execute specific code paths based on boolean expressions. Example: if (temperature > threshold) { activateCooling(); }
  • switch/case: Multi-way branching for handling multiple discrete conditions efficiently, often used for state machine implementations.
  • for loops: Deterministic iteration for known iteration counts, ideal for traversing arrays or executing a fixed number of operations.
  • while loops: Conditional iteration that continues as long as a condition remains true, commonly used for waiting on hardware flags or continuous monitoring.
  • do-while loops: Executes at least once before checking the condition, useful for initialization sequences that must run before condition evaluation.

Functions and Modular Programming

Functions decompose complex programs into reusable, testable units. In embedded systems, modularity is not merely aesthetic — it facilitates code reuse across projects, simplifies debugging, and enables team collaboration.

  • Function Declaration: The function signature specifying return type, name, and parameters — the contract between the function and its callers.
  • Function Definition: The actual implementation containing the executable logic.
  • Function Call: Invoking the function with specific arguments to achieve a desired outcome.
  • Return Values: Functions can return computed results to the caller, enabling composition of complex operations from simpler building blocks.
  • Scope: Variables declared within functions are local — they exist only during function execution — reducing unintended side effects.

Pointers and Memory Management

Pointers represent the most powerful and dangerous feature of C. A pointer is a variable that stores a memory address — the location of another variable in RAM. Through pointers, we can:

  • Access hardware registers mapped to specific memory addresses
  • Pass large data structures efficiently without copying (by passing the address)
  • Implement dynamic data structures like linked lists and ring buffers
  • Manipulate individual bits within memory words

However, pointer misuse — dereferencing null pointers, accessing freed memory, or pointer arithmetic errors — constitutes the leading cause of embedded software failures.

Bitwise Operations

Embedded systems frequently require manipulation of individual bits within hardware registers. Bitwise operators provide this capability:

  • & (AND): Clears specific bits: reg = reg & 0xF0; clears the lower four bits
  • | (OR): Sets specific bits: reg = reg | 0x01; sets the lowest bit
  • ^ (XOR): Toggles specific bits: reg = reg ^ 0xFF; inverts all bits
  • ~ (NOT): Inverts all bits in a value
  • << (Left Shift): Multiplies by powers of two or positions bits for masking
  • >> (Right Shift): Divides by powers of two or extracts bit fields

Assembly Language

Assembly language represents the lowest tier of human-readable programming — each instruction corresponds directly to a single machine code instruction that the processor executes. While C abstracts hardware details for programmer convenience, Assembly grants absolute, granular control over every processor operation.

Registers and Memory Addressing

Registers are the fastest memory locations within a processor — small storage areas built directly into the CPU silicon. The ARM Cortex-M4 processor (used in the Raspberry Pi Pico) contains:

  • General-Purpose Registers (R0-R12): Twelve registers for storing temporary values and function arguments. R0-R7 are accessible to all code; R8-R12 are banked in privileged modes.
  • Stack Pointer (SP): Tracks the current position in memory where function call frames are stored. Two variants exist: Main Stack Pointer (MSP) for privileged code and Process Stack Pointer (PSP) for unprivileged applications.
  • Link Register (LR): Stores the return address when a function is called, enabling the processor to resume execution at the correct location upon function return.
  • Program Counter (PC): Contains the address of the next instruction to be executed — automatically incremented after each instruction.
  • Program Status Register (PSR): Contains flags indicating arithmetic results (zero, negative, carry, overflow) and current processor mode.

Instruction Sets

The ARM Cortex-M4 implements the Thumb-2 instruction set, a hybrid of 16-bit and 32-bit instructions that balances code density with performance:

  • Data Processing: ADD, SUB, MUL, DIV, AND, ORR, EOR, MOV, CMP — arithmetic and logical operations
  • Load/Store: LDR, STR, LDM, STM — transferring data between registers and memory
  • Branch: B, BL, BX, BLX — controlling program flow through jumps and function calls
  • Special: MRS, MSR — accessing special registers like the PSR and control registers

Assembly Syntax and Mnemonics

Assembly instructions follow a consistent pattern: Mnemonic{condition}{flags} Destination, Source1, Source2

Example: ADDS R0, R1, R2 adds R1 and R2, stores the result in R0, and updates the flags. The S suffix indicates flag update.

Interfacing with Hardware Peripherals

Through Assembly, we can directly manipulate memory-mapped peripherals — hardware components (UART, SPI, I2C, GPIO) whose control registers appear as memory addresses to the processor. Writing to these addresses triggers hardware actions:

  • Writing to a GPIO output register toggles pin voltage levels
  • Reading a UART status register reveals whether data is available
  • Configuring a timer register initiates countdown operations

Programming for Microcontrollers

Embedded programming differs fundamentally from desktop or web development. Microcontrollers impose strict constraints that shape every design decision:

Hardware Constraints

  • Limited RAM: Typical embedded MCUs offer 2KB to 512KB of RAM — insufficient for naive memory allocation strategies
  • Limited Flash: Program storage ranges from 8KB to 2MB, demanding efficient code and avoidance of unnecessary libraries
  • No Operating System: The program runs directly on hardware — there is no memory protection, process isolation, or system calls
  • Fixed Peripherals: Hardware peripherals are fixed at manufacturing — software must adapt to their capabilities

Optimizing for Limited Resources

  • Static Allocation: Prefer fixed-size arrays over dynamic allocation to avoid heap fragmentation
  • Bit Fields: Pack multiple boolean flags into single bytes to conserve memory
  • Inline Functions: Eliminate function call overhead for small, frequently-called routines
  • Const Qualifier: Place constant data in flash memory rather than RAM

Working with Peripherals

  • GPIO (General-Purpose Input/Output): Configurable pins that can read external signals (input) or drive voltage levels (output)
  • Timers: Hardware counters that generate precise timebase signals for PWM, input capture, or scheduling
  • Interrupts: Asynchronous events that pause main program execution to handle time-critical events
  • Communication Peripherals: UART, SPI, I2C for interfacing with sensors, displays, and other microcontrollers

Fundamental Rules in Embedded Systems

Embedded development is governed by principles that distinguish it from general software engineering:

Memory Efficiency

Every byte has a cost. Embedded developers must:

  • Calculate maximum memory usage before deployment
  • Use appropriate data types — avoid 32-bit integers when 8-bit suffices
  • Implement custom memory pools for predictable allocation
  • Monitor stack usage to prevent overflow

Real-Time Constraints

Many embedded applications have strict timing requirements:

  • Hard Real-Time: Missing a deadline causes system failure (e.g., airbag deployment)
  • Soft Real-Time: Missing deadlines degrades performance but does not fail (e.g., audio playback)
  • Interrupt latency must be bounded and predictable
  • Algorithm complexity must be analyzed for worst-case execution time

Hardware Abstraction Layers

Abstraction separates hardware-specific code from application logic:

  • Hardware Abstraction Layers (HAL) provide consistent APIs across MCU families
  • Board Support Packages (BSP) adapt HAL to specific hardware configurations
  • This separation enables portability — the same application can run on different hardware

Debugging and Testing

Embedded debugging presents unique challenges:

  • JTAG/SWD: Hardware debug interfaces for stepping through code and inspecting registers
  • Serial Output: printf-style debugging via UART when hardware debuggers are unavailable
  • Unit Testing: Testing individual functions in isolation from hardware
  • Hardware-in-the-Loop: Testing with actual peripherals connected

Week 2 provided us with the essential programming competencies required to develop firmware for embedded devices. The dual mastery of C's abstraction and Assembly's granularity equips us to bridge the conceptual chasm between algorithmic intent and physical actuation — the fundamental objective of embedded systems engineering.
